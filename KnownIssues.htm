Known Issues for XSL ISAPI Filter 2.0

Server-side XSL formatting for multiple device-types

May 2000

This is a companion article to the XSL ISAPI Filter 2.0 Server-side XSL formatting for multiple device-types article.  This article enumerates and discusses known issues that may be of interest to a server-administrator or content-developer using the XSL ISAPI filter.

Various Tips and Tricks are also discussed here.

PASP Restrictions

The PASP files used to generate XML dynamically and then pass into the XSL ISAPI pipeline have a few additional restrictions above and beyond those imposed by ASP itself.

         The ASP include mechanism (<!--#include File="includeMe.inc"-->) is not supported in PASP files.

         Request.BinaryRead and Response.BinaryWrite are not supported.

         The XSL ISAPI pipeline introduces an additional HTTP server variable (HTTP_SSXSLSRCFILE) that is not present without the pipeline.

         If the <@LANGUAGE > directive is in a PASP file, it must be the absolute first line of that file, before any <OBJECT> declarations or even any carriage returns.

         XSL ISAPI works by preprocessing PASP files into actual ASP files.  PASPs are only re-processed by the system if they have a newer timestamp than the ASP thats generated from it.  Therefore, whenever copying over PASP files, be sure that their timestamp is updated.

         Its often desirable to be able to populate an XML DOM document object from ASP scripts.  This is typically done via the load method.  One should bear in mind that doing a load on a fully-qualified URL is not currently supported by XML.  (More accurately, its not supported by WinInet, which is used by the load method.) That is, something like

            doc.load(http://www.microsoft.com/xmlSources/myXml.xml)

is not supported.  Loading from a local source on the machine is supported:

            doc.load(Server.MapPath(/myVdir/myXml.xml))

See the Knowledge Base article at http://support.microsoft.com/support/kb/articles/Q237/9/06.ASP for more information.

         Due to the way PASP files write into an XML DOM document, its not currently feasible to include a DOCTYPE node in a PASP file.  It simply wont generate the proper XML.  Therefore, theres currently no way to take advantage of stylesheet selection based on DOCTYPE for dynamically generated XML content.  (All works fine for static XML.)  The workaround is to have the generated XML reference separate configuration files instead.

File Location Issue

         XSL ISAPI does not recommend the use of a fully-qualified URL as the path to a configuration or stylesheet file. This is due to known issues with the use of WinInet from a server.

See the Knowledge Base article at http://support.microsoft.com/support/kb/articles/Q237/9/06.ASP for more information.

General Server Administration Issues

         Making a PASP file or an XML file the default document will not work.  Instead, have a default.asp that does a Response.Redirect to the desired PASP or XML file.

         Due to the way XSL ISAPI maps URLs, the logging information contains URLs that are hit after the mapping (namely RedirectorXML.asp and RedirectPASP.asp).  If so desired, the original requests can be logged by modifying those Redirector*.asp files to write out the original path to a log file.

         XSL ISAPI will ignore requests for .xml files in virtual directories that have an ISAPI extension registered for the .xml or * extensions.  Thus, such files will be passed directly on to the extension without going through XSL ISAPI.  The Microsoft SQL Server XML Technology Preview is an example of an ISAPI extension that takes over the * extension in the directories it operates in.

Note also that XSL ISAPI  1.1 registers .xml so XSL ISAPI 1.1 must be uninstalled in order for XSL ISAPI 2.0 to function properly.

Security-related Issues

         Because of the way the XSL ISAPI filter redirects URL requests to a known ASP file (/xslisapi/RedirectorXML.asp), and the server-side objects invoked by that ASP file use Win32 filesystem functions to access the original XML file, only filesystem-based permissions (ACLs) on the XML files will be honored.

XSL Authoring Information

The following comments are not specific to XSL ISAPI, and are general issues with developing using XSL.  However, they are relevant in the context of rendering for cell-phone markup languages.

         The WML cellphone markup language is very sensitive to stray whitespace.  Care must be taken in forming XSL that doesnt introduce such whitespace.  For instance, this XSL emits WML:

    <xsl:attribute name="onpick"><xsl:value-of select="link"></xsl:attribute>

does the right thing and inserts no additional whitespace.  However:

    <xsl:attribute name="onpick">
       <xsl:value-of select="link">
   </xsl:attribute>

will insert whitespace between the attribute name and value, and thus produce non-compliant WML.

Tips and Tricks

         The installation instructions discuss adding the contents of browscap-add.ini to the systems browscap.ini file.  For situations where you want to choose stylesheets solely based upon target-markup language, you may want to update your browscap.ini to include a content-type property for every browser youre interested in serving, and then basing your <device> clauses on the content-type property.

         Chaining of stylesheets may be used as a technique to pull common transformations out of multiple stylesheets into a single stylesheet thats always used in part of the chain.  This can be useful for consistent insertion of navigation and legal information.

         When dynamically generating XML via PASP files, it can be a useful debugging technique to look at the generated XML directly rather than processing it through XSL.  This can be done simply by removing the href and server-href attributes from the xml-stylesheet processing instruction, forcing the generated XML to be sent directly to the client browser (which, in the case of Internet Explorer, will do a nice job of structuring that XML for viewing).

Microsoft Windows NT 4 Restrictions

The RedirectorPASP.asp file for NT4 relies on Response.Redirect which creates the following limitations when handling .pasp files:

         Using Response.Redirect will result in additional server to client to server roundtrip. This will impact performance and may be particularly noticeable on low bandwidth connections.

         All client browsers will need to support automatically handling a 302 redirection.

         Postfields sent to the .pasp URL via the HTTP POST method (such as HTML form fields) will be ignored. 

         The URL displayed in the client browser will not be the same as the browsed-to URL (for example, __foo.asp will be displayed instead of foo.pasp.)
