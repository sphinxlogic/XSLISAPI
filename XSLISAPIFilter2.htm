XSL ISAPI Filter 2.0

Server-side XSL formatting for multiple device-types

May 2000

Introduction

Web browsing devices are becoming increasingly more widespread and varied.  Witness the variety of Internet-enabled cellphones, handhelds, voice-enabled browsers, and television and game-console based browsers becoming available.  Whats more, different devices within a form-factor may have different markup languages that it interprets.

All of this makes the job of the content developer that wants to optimize presentation on each device considerably more complex.  XML and XSL provide pieces of a solution to this dilemma, whereby content developers can provide their data in XML form and then provide XSL stylesheets for each different device that he or she is interested in supporting.  The XSL is used to filter and transform the XML for presentation on the appropriate device.  However, there are missing pieces to this story, and the XSL ISAPI Filter is a tool that helps bridge this gap.

Motivation

The XSL ISAPI Filter is motivated primarily by these concerns:

         Not all client browsers (including more and more browsers for small and wireless devices) support XSL.

         Choosing amongst stylesheets based upon browser device characteristics, and then executing them on the server, is difficult.

         Dynamically generating XML via ASP pages and having the result transformed by XSL on the server is a difficult and cumbersome process.

Features

The Microsoft XSL ISAPI Filter simplifies the task of performing server-side XSL transformations. It features:

1.      Automatic execution of XSL stylesheets on the server, choosing alternate stylesheets based on browser device characteristics and/or XML document type

2.      Pipelining of ASP-generated XML into XSL stylesheets

3.      Stylesheet caching for improved server performance.  Includes caching of precompiled stylesheets via the XSLTemplate object on systems with this object installed.

4.      Chaining execution of multiple stylesheets

5.      Multiple, configurable output MIME types

6.      Post-processing support for cellphone markup languages HDML, WML 1.0, and WML 1.1

7.      Ability to specify output encodings and character-sets

8.      Device-customizable error messages

9.      Configurable pass-through of the XML for subsequent client-side XSL processing

This product is being made available as a technology preview with full source code.  It is a follow-on to the XSL ISAPI Extension 1.1 release.

Whats New

A number of the above features are new since XSL ISAPI Extension 1.1, including pipelining of ASP-generated XML, a richer device-characteristics vocabulary, configurable MIME types, chaining of XSL stylesheets, use of the XSLTemplate object, post-processing support, and custom error messages that can be formatted based upon device characteristics.

 

System Requirements

The XSL ISAPI Filter is targeted for use on Microsoft Windows 2000 Server and Microsoft Windows 2000 Advanced Server running with MSXML 2.5 or MSXML 3.0.  Other configurations that will work, but havent been exhaustively tested, include:

1.      NT4 with IIS 4.0 This will work, however, see the Known Issues document for additional limitations when using dynamically generated XML and a file must be substituted during the installation process.

2.      MSXML 2.6, and versions after 3.0.

Installation

The distribution should be downloaded and the files expanded onto your hard drive.  Files referenced below will be mentioned relative to the place at which theyre downloaded.

Minimal Required Installation steps

The following steps are required for any usage of XSL ISAPI: 

1.      Register the xslisapi2.dll via regsvr32 xslisapi2.dll.  A dialog box indicating success should be displayed.

2.      Using the IIS Administration tool, right-click on Default Web Site, select Properties, select the ISAPI Filters tab, select Add, type the filter name (e.g., XSLISAPI2), and browse to xslisapi2.dll for the executable.  Click OK, then OK again. 

3.      Verify that the filter is properly installed by going back to the ISAPI Filters tab and seeing that theres a green up-arrow next to the filter name.

4.      Create an xslisapi virtual directory (note this is xslisapi and not xslisapi2) on your web server. 

a.   Launching the Create New Virtual Directory wizard by right clicking on Default Web Site, and selecting New Virtual Directory.

b.   Name the virtual directory xslisapi

c.   Make this virtual directory point to the directory on your machine that you expanded the distribution into.  Be sure that this directory has the following files: RedirectorXML.asp, errorConfig.xml, errorHTML.xsl, and errorWML11.xsl.

d.   Be sure the Read and Run scripts checkboxes are checked on the permissions dialog.

Additional Steps for Dynamic XML

If you want to have dynamically generated XML (via ASP files) processed through XSL ISAPI, follow these additional steps (note these steps have already been done for the included samples that perform dynamic XML generation):

For each IIS directory that will contain .pasp files (explained in ASP-produced XML content section), you need to

         Copy RedirectorPASP.asp from the XSLISAPI2 distribution into the directory that the .pasp files will be in.

         Copy global.asa from the XSLISAPI2 distribution into that same directory (or anywhere higher up in the directory structure of the IIS Application, since all global.asas up the hierarchy are visited.) 

         If theres already a global.asa file where you want to put it, youll need to merge the contents of the two.

         Lastly, XSL ISAPI 2.0 will write files into the same directory as the .pasp files are in, thus permissions need to be set accordingly.  If the directory has restricted permissions, then write and modify access must be explicitly granted for the IWAM_machinename user.  (Note that this is only the case if the IIS Application that the directory is part of has Application Protection set to Medium or High).

Note: There are two RedirectorPASP.asp files that ship with XSL ISAPI 2. The default RedirectorPASP.asp (in the main distribution file) works on Windows 2000 Server but will not work on NT 4.  Both of the RedirectorPASP.asp files contain comments that identify their corresponding operating system. 

These steps must be performed if you want to dynamically generate XML on NT 4.

         Open the NT4 folder in the distribution.

         Copy the RedirectorPASP.asp file from the NT4 folder and replace all copies of the Windows 2000 RedirectorPASP.asp file (including those in the Samples directories).  A copy of the correct RedirectorPASP.asp file must be in all directories that will contain .pasp files.

         In the IIS Admin utility, perform the following steps:

o        Bring up properties page of Default Web Site or the corresponding Virtual Directory.

o        Go to Home Directory tab, click the button Configuration...

o        Go to App Options tab, make sure Enable buffering is checked.

Additional Steps for Extended Browser Information

Follow the steps below for adding information about additional browsers to IISs browser information database, browscap.ini:

         Add the contents of browscap-add.ini from the XSLISAPI2 distribution into your %SYSTEMROOT%\system32\inetsrv\browscap.ini.  This will add identifying information about a number of cell-phone browser simulators.  This doesnt need to be done if youre not planning on using the browsers specified in browscap-add.ini.  Add the contents just above the Default Browser section.

         Add sections for whatever other browsers you might be using to the %SYSTEMROOT%\system32\inetsrv\browscap.ini file, along with identifying information that youll want to use in your configuration files (configuration files are discussed in a subsequent section).

Verify Installation

After all the installation steps have been performed, you should stop and restart the IIS service through the IIS Administration tool.

To verify that the installation was done properly, use an HTML browser to visit http://localhost/xslisapi/samples/install/installtest.xml.  This should result in text indicating that the filter is installed properly.  If this file doesnt exist, its likely because you pointed the virtual directory to a directory other than the one you expanded the XSLISAPI distribution into. 

Also visit http://localhost/xslisapi/samples/install/installtest.pasp.  This tests dynamic XML generation, and should result in text displaying that the installation is valid.

If either of these doesnt work, look back at the installation steps to ensure that you didnt miss any.

 

Usage

Stylesheet Selection

Once the XSL ISAPI filter is installed, the filter will examine all requests for a URL with a .xml extension.  XML files that have a processing instruction of the form:

 

<?xml-stylesheet type=text/xsl server-config=myConfig.xml
                                 href=clientSS.xsl ?>

 

are selected for processing by XSL ISAPI.  The configuration file specified in the server-config attribute will be loaded, and it will be used to determine what stylesheet(s) to apply to the XML file.  For example, myConfig.xml might be:

 

<?xml version="1.0" ?>

<server-styles-config>

  <device target-markup="WML1.1">

    <stylesheet href="wml11Style.xsl"/>

  </device>

  <device browser="IE" version="5.0">

    <stylesheet href="ie50Style.xsl"/>

  </device>

  <device browser="IE" version="3.0">

    <stylesheet href="ie30Style.xsl"/>

  </device>

</server-styles-config>

 

This configuration file chooses between three XSL stylesheets to be applied, based upon the device characteristics of the requesting browser.

Device characteristics & browscap.ini

Each <device> element in the configuration file contains an arbitrary number of attributes.  These attributes are used to determine if the stylesheets specified as child elements of the <device> element will be used.  XSL ISAPI uses the ASP BrowserType object to compare each attribute with the corresponding device property in the %SYSTEMDIR%\system32\inetsrv\browscap.ini database.  The string comparison performed is case-sensitive.  If all of a <device> elements attributes are matched, the <device> elements child <stylesheet> elements are selected.  Some more rules governing this selection are:

         The <device> elements are evaluated from top to bottom, with the first match winning.

         If a property doesnt exist in browscap.ini for the requesting browser, a value of undefined is used.

         A <device> element without any attributes is considered an automatic match, and its stylesheet child elements are selected.

         If no <device> elements match, the XML is sent, untouched, to the browser, with a content-type of text/xml.  In the above example, that would result in the clientSS.xsl stylesheet being applied on the client machine.

         If there is a <device> match, but that <device> element doesnt contain any <stylesheet> child elements, the XML is sent, untouched, to the browser.[1]

Stylesheet chaining

Multiple XSL stylesheets may be chained together simply by specifying them all as children of a <device> node.  For example,

 

<device browser="IE" version="5.0">

  <stylesheet href="removeApples.xsl"/>

  <stylesheet href="sortByPrice.xsl"/>

  <stylesheet href="ie50Style.xsl"/>

</device>

 

would first apply removeApples.xsl to the incoming XML, pass the result on to sortByPrice.xsl, and pass that output on to ie50Style.xsl.  Note that all but the last stylesheet must be an XML to XML transformation, since the input to the next stylesheet is expected to be XML.

There is currently a maximum of 64 chained stylesheets.  If necessary, this is easily modified in the source code.

content-type specification

Different stylesheets may produce output that should be sent along with a different MIME type.  Choosing the MIME type proceeds as follows:

         The chosen <device> element is searched for a <content-type> child, as in:

<device >
  <content-type type="text/myType" />
  <stylesheet href=".xsl" />
</device>

if found, this value is used.

         The browscap.ini database is consulted for the content-type property.  If it exists, it is used.

         Otherwise, a content-type of text/html is used.

Doctype-based selection

All of the above examples describe stylesheet selection based solely upon device characteristics.  It can also be useful to choose the stylesheet based upon the nature of the XML document being processed.  To that end, XSL ISAPI supports stylesheet selection based upon the doctype of the incoming XML document.  Here is a sample configuration file:

 

<?xml version="1.0" ?>

<server-styles-config>

 

  <device target-markup="WML1.1">

    <doctype name=invoice>   

      <stylesheet href="wml11Invoice.xsl"/>

    </doctype>

    <doctype name=productDescrip, descrip>

      <stylesheet href=wml11Descrip.xsl/>

    </doctype>

  </device>

 

  <device browser="IE" version="5.0">

    <doctype name=invoice>   

      <stylesheet href="ie5Invoice.xsl"/>

    </doctype>

    <doctype name=productDescrip, descrip>

      <stylesheet href=ie5Descrip.xsl/>

    </doctype>

  </device>   

 

</server-styles-config>

 

This configuration file will, for a WML 1.1 device, choose wml11Invoice.xsl for XML documents with a doctype property of invoice, and wml11Descrip.xsl for documents with a doctype of either productDescrip or descrip.  The IE section behaves similarly.

As is the case with <device>, <doctype>s are processed top-down, and a <doctype> without a name attribute will always be selected.  Again, the comparison performed is case-sensitive.

The direct children of a <device> element cannot mix <doctype> and <stylesheet> elements.  Thus, the following is illegal:

 

<?xml version="1.0" ?>

<server-styles-config>

  <device target-markup="WML1.1">

 

    <doctype name=invoice>   

      <stylesheet href="wml11Invoice.xsl"/>

    </doctype>

 

    <!-- Illegal to have stylesheet at this level given

         that we have a doctype sibling -->

    <stylesheet href=wml11Descrip.xsl/>

 

  </device>

</server-styles-config>

 

Encoding and character set issues

Both the code page encoding and the character set can be specified in the configuration file and via browscap.ini.  The code page is used on the server during XSL transformation, and the character set is used by the client browser to render properly.

XSL ISAPI first looks in the chosen <device> node for charset and encoding, looking for formulations like:

  <device >
    <output encoding="codepageName" charset="charsetName" />
    <stylesheet href=".xsl" />
  </device>

If encoding is not found, the encoding property from the browscap.ini database is used if it is defined.  The same is done for charset.

Other rules followed:

         Both character set and the encoding properties are optional in the configuration file and the browscap.ini database.

         If both a code page and charset are provided, XSL ISAPI validates the code pages for "recognized" charsets and returns an error if they are incorrect (e.g. the char set does not match code page).  If the charset is not recognized we assume that the code page is valid. XSL ISAPI will use the valid code page for the transformation and the charset will be sent to the client.

         If no code page and no charset are provided, UTF-8 will be used for the transformation and the string "UTF-8" will be used for charset in HTTP header sent to the client.

         If a charset is provided and no code page is provided, XSL ISAPI will look up the charset in an internal mapping table (in charset.cpp) to determine the code page. The mapping table will only contain known charset strings for IE 4 and IE 5.  If the charset is not found in the mapping table an error will be returned specifying that a code page is required.

         If a code page is provided and no charset is provided, the code page will be used for the transformation and will set the charset as the same string as the code page string.

 

ASP-produced XML Content

One of XSL ISAPI 2.0s most powerful features is the ability to use ASP to dynamically create XML data that is then pipelined through server-side XSL formatting in a manner identical to the static XML content pipeline described above. 

In order for an ASP script to be processed correctly, it needs to be stored in a file with a .pasp extension and it must result in valid XML.  For instance, consider the following ASP script stored in the file time.pasp:

 

<?xml version="1.0" ?>

<?xml-stylesheet type=text/xsl server-config=myConfig.xml ?>

<currentTime>

   <%= Now %>

</currentTime>

 

When time.pasp is browsed to, XML data like the following will be generated and then fed into whatever stylesheets are determined by the contents of myConfig.xml:

 

<?xml version="1.0" ?>

<?xml-stylesheet type=text/xsl server-config=myConfig.xml ?>

<currentTime>

   4/28/2000 2:49:59 PM

</currentTime>

 

This simple mechanism provides a very powerful, flexible way to dynamically generate and then format XML data, all on the server.

Details and Restrictions

Because of the way XSL ISAPI 2.0 works, not everything that is expressible in ASP is supported in a .pasp file.  The following ASP features are not supported in .pasp files:

         Use of server-side objects in the ASP that write directly to the Response object.  Such objects should instead return strings into the ASP script, and the ASP should then pass these onto the Response object via either Response.Write or the <%= %> syntax

         Inclusion of ASP files via the <--#include File=--> syntax

         See the companion KnownIssues.htm document for additional, less significant restrictions and limitations specific to NT 4 implementations.

Also, the section of the Installation Instructions titled Additional Steps for Dynamic XML needs to be followed carefully.

Error Handling

XSL ISAPI 2.0 provides highly configurable error handling to, among other uses, add customized processing for display of error messages on a variety of devices.

An example chunk of XML representing an error is included here:

 

<?xml-stylesheet type=text/xsl

                 server-config=/xslisapi/errorConfig.xml ?>

<error>

   <status-code>404 Not Found</status-code>

   <url>someURL.xml</url>

   <info>Some more information goes here...</info>

</error>

 

This XML is generated by the XSL ISAPI code and is processed through XSL ISAPI just as all other XML content is, in this case using the specified /xslisapi/errorConfig.xml configuration file.  This configuration file (and its constituent stylesheets) can be modified to both alter the rendering of the error and to specialize rendering for new devices.

Post-processing

XSL ISAPI 2.0 supports a limited form of post-processing of the stream thats generated by applying the XSL stylesheets specified in the configuration files.  This post-processing is useful for achieving results that are not expressible through XSL and yet are required for the targeted markup language.  The post-processing handles three situations:

         Removing closing slashes.  HDML requires that the ACTION element be formed as <ACTION> and not <ACTION/>.  This is not well-formed XML, and cannot be generated by XSL.  This is true for other elements as well.

         Escaping symbols.  The symbol $ must be escaped for acceptance in an HDML or WML browser.  In WML, it must become $$.  In HDML, it must become &dol;

         Replacing entities.  HDML and WML need to be able to have a $ emitted into their stream to allow for variable dereferencing.  If the XSL produces the entity &var;, the post-processor will replace this with $ for HDML and WML.

 

The post-processing that occurs is table-driven, and may be modified at XSLISAPI installations by modifying the relevant source file ProcessingStream.cpp.  The variable g_aMimeToStreamTypeMap maps from target MIME type to a table of post-processing parameters that will be used for that MIME type.

Caching

XSL ISAPI 2.0 supports a simple caching scheme for in-memory caching of both configuration files and XSL stylesheets.  The caching strategy is a variant of a least-recently-used cache and works as follows:  Once every n minutes, any element in the cache that hasnt been accessed in over n minutes is discarded.

The value of n may be set in a file located at /xslisapi/masterConfig.xml that contains the following XML, with the <config> element at the top level of the document:

 

<config>

       <cache cleanup="1440"/>

</config>

 

The above sets the cleanup time for 1440 minutes, or 24 hours.  Setting this value to 0 results in disabling caching entirely.  The default value is 60 minutes.

On systems where the XSLTemplate object is available (via msxml3.dll), stylesheets will be precompiled and cached as XSLTemplate objects for improved performance.  Otherwise, they are cached as XMLDOM documents.

XSL Version Information

XSL ISAPI 2.0 will successfully process XSL stylesheets that are compatible with either msxml.dll or, if its installed on the system, msxml3.dll (including the XPath/XSLT features of msxml3.dll).

Backwards compatibility with XSL ISAPI 1.1

This section is only of interest to current users of XSL ISAPI 1.1 that want to port their work over to XSL ISAPI 2.0.

XSL ISAPI 2.0 is substantially backwards compatible with XSL ISAPI 1.1:

         The contents of the XSL ISAPI 1.1s configuration file will behave as they did for XSL ISAPI 1.1, but the file itself must now be located in the xslisapi virtual directory at \xslisapi\masterConfig.xml.

         Settings in masterConfig.xml are overridden by any setting destined for XSL ISAPI 2.0.  For example, if an <?xml-stylesheet?> processing instruction contains a server-config attribute pointing to an XSL ISAPI 2.0 configuration file, this will be used in favor of any attributes that match <client> elements in the masterConfig.xml file.

         While the error schema is the same in 2.0 as in 1.1, 2.0 does not look for an errors.xsl file, due to the more general error handling mechanism present.  However, the errors.xsl file can be specified as a device stylesheet in the errorConfig.xml file if desired.

         Encoding will work when using version 2.0 with XSL ISAPI 1.1 config files, if the charset and code page encoding worked under version 1.1.  The limitations of code page and charset compatability that currently exist with 1.1 config files will continue to persist, however if it worked in version 1.1 it will work in version 2.0.

Samples

The XSL ISAPI distribution contains a number of samples.  If the Installation instructions above have been followed, these can be reached at http://localhost/xslisapi/Samples.

Each sample presents a minimal demonstration of a single feature of XSL ISAPI, as follows:

Sample A

The most basic demonstration of choosing stylesheets based upon device characteristics, and applying the chosen stylesheet to a static XML document.

Sample B

Demonstrates chaining of XSL stylesheets.

Sample C

Demonstrates selection of stylesheets based upon both device characteristics and document type.

Sample D

Demonstrates the use of ASP to dynamically generate XML and then have a result pass through stylesheet selection similar to Sample A.

Sample E

Demonstrates the use of a character set encoding.

Sample F

A more involved sample that dynamically generates XML by breaking up a large XML document into chunks small enough to be held by the target device.  This is motivated by the fact that cell-phone browsers typically have limited memory capacity.
